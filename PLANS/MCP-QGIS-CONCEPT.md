# MCP + QGIS Preconcept (проект 4114)

Версия документа: 0.1  
Дата: 2026-02-11

## 1. Контекст и цель

Цель: построить MCP-сервис, через который модель (Codex / Claude Code / Antigravity) выполняет команды в QGIS на естественном языке.

Ключевой бизнес-сценарий:

- деление исходного участка на меньшие;
- резервирование дороги между участками;
- прокладка/проверка коммуникаций;
- точечные правки геометрии (в т.ч. смещения границ на заданные метры);
- сравнение вариантов планировки и выбор лучшего.

## 2. Что важно зафиксировать сразу

Под каждый частный кейс отдельный MCP-метод не нужен.

Нужна архитектура:

- небольшой набор универсальных гео-примитивов;
- планировщик, который превращает текстовую задачу в цепочку примитивов;
- единый валидатор кадастровых ограничений;
- безопасное применение изменений (транзакции, rollback, Git snapshot).

Идея: не “1000 команд”, а “один движок + правила + примитивы”.

## 3. Оценка базы `qgis_mcp` (на 11 февраля 2026)

Короткий вывод:

- как foundation: **6/10**;
- закрытие потребностей “из коробки”: **35–45%**.

Что уже хорошо:

- связь MCP <-> открытый QGIS;
- базовая работа с проектом и слоями;
- запуск Processing через `execute_processing`;
- fallback через `execute_code` (PyQGIS).

Что не закрыто для целевого workflow:

- нет orchestration-слоя “текст -> безопасный план действий”;
- нет встроенных кадастровых guardrails (CRS/топология/площади/доступ);
- нет вариативного проектирования и автоматического сравнения вариантов;
- нет встроенных Git-снимков “до важных операций”;
- нет формального пошагового сценария для пользователя без программирования.

Инженерные риски текущей базы:

- баг проверки соединения (`.sock` vs `.socket`);
- зашитый порт `9876`;
- один активный клиент (`listen(1)`);
- нет лицензии в репозитории;
- нет релизов;
- нет CI/тестов в `main`;
- есть полезный, но незамерженный PR по стабильности/сериализации.

Вывод:

- использовать как форк-основу можно;
- для production-уровня нужен отдельный “кадастровый слой” поверх.

## 4. Продуктовый принцип

Один и тот же набор примитивов покрывает разные фразы:

- “раздели на 4 + дорога + коммуникации” -> длинная цепочка примитивов;
- “сдвинь общую границу A/C на 3 м” -> короткая цепочка примитивов.

Разница только в плане, а не в “новом методе под каждый кейс”.

## 5. Целевая архитектура MCP

## 5.1 Слои

1. Client Layer  
Codex / Claude Code / Antigravity отправляют запросы в MCP.

2. Intent Layer  
Разбор намерения, параметров и ограничений пользователя.

3. Planner Layer  
Построение формального плана (Plan IR) из универсальных примитивов.

4. Execution Layer  
Исполнение шагов через PyQGIS + Processing (`native:*`) в транзакционном режиме.

5. Validation Layer  
Кадастровые проверки до/после шага и перед commit.

6. Versioning Layer  
Git snapshot, журнал изменений, возможность rollback.

7. Artifact Layer  
Экспорт результатов (GeoPackage/GeoJSON/QGS), отчеты и рендеры.

## 5.2 Режимы исполнения

- Mode A: работа с открытым QGIS (основной интерактивный режим).
- Mode B: headless-пайплайн (часть операций через `qgis_process`).

Рекомендация:

- начать с Mode A;
- затем добавить Mode B для пакетных задач/сервера.

## 5.3 План как промежуточный контракт (Plan IR)

Пример структуры:

```json
{
  "plan_id": "plan-2026-02-11-001",
  "project": "4114/QGIS/plot-4114.qgs",
  "crs": "EPSG:32637",
  "steps": [
    {"op": "fix", "layer": "parcel_src"},
    {"op": "split", "input": "parcel_src", "splitter": "road_axis"},
    {"op": "difference", "input": "lots_tmp", "overlay": "road_polygon"},
    {"op": "snap", "input": "lots_tmp2", "reference": "lots_tmp2", "tol_m": 0.02},
    {"op": "validate", "ruleset": "cadastre_default"},
    {"op": "commit"}
  ]
}
```

## 6. Универсальные примитивы (ядро)

Минимальный набор операций:

- `split`
- `buffer`
- `difference`
- `intersection`
- `snap`
- `fix`
- `move` / `translate`
- `merge`
- `clip`
- `validate`
- `commit`
- `rollback`

Опционально:

- `route` (для коммуникаций/доступа);
- `variant_clone` (для ветвления вариантов).

## 7. Предлагаемый набор MCP tools (не частные, а универсальные)

1. `project_open(path)`  
2. `project_state()`  
3. `layer_catalog()`  
4. `intent_to_plan(intent_text, constraints)`  
5. `plan_preview(plan)`  
6. `plan_validate(plan, ruleset)`  
7. `plan_execute(plan, dry_run=false)`  
8. `topology_validate(layer_ids, ruleset)`  
9. `variant_create(base_state, name)`  
10. `variant_compare(variant_ids, metrics)`  
11. `git_snapshot(message)`  
12. `export_result(targets, format, path)`

Ключевая идея:

- high-level команда всегда идет через `intent_to_plan` + `plan_validate` + `plan_execute`.
- не добавляем “метод на каждый кейс”.

## 8. Guardrails (обязательные проверки)

Правила по умолчанию `cadastre_default`:

- CRS проекта и рабочих слоев согласованы;
- рабочая CRS метрическая (для дистанций в метрах);
- геометрии валидны;
- нет самопересечений;
- нет недопустимых overlap/gap между смежными участками;
- площади участков внутри допустимых ограничений;
- дорожный коридор непрерывен;
- для каждого участка есть доступ к дороге (если включено правилом);
- коммуникационные коридоры не нарушают заданные отступы.

Пороговые значения (tolerance) задаются конфигом проекта.

## 9. Транзакционная модель

Для каждого `plan_execute`:

1. `tx.begin`
2. выполнение шагов с промежуточными проверками
3. финальная валидация
4. `tx.commit` при успехе
5. `tx.rollback` при ошибке

Перед важными изменениями:

- `git_snapshot("before <plan_id>")`

## 10. Формальный workflow для пользователя без программирования

Мастер из простых шагов:

1. Что хотите сделать?
2. С каким участком/слоем?
3. Какие ограничения? (минимальная площадь, ширина дороги, отступы)
4. Сколько вариантов построить?
5. Автоприменять или сначала показать план?

Система генерирует:

- понятный черновик плана;
- список проверок;
- предупреждения рисков;
- кнопку “Применить” или “Исправить параметры”.

## 11. Как один движок покрывает разные команды

Сценарий A:

- команда: “раздели на 4, дорога 6 м, учти коммуникации”
- результат: план из ~10–20 шагов (`split/buffer/difference/snap/fix/validate`).

Сценарий B:

- команда: “сдвинь границу A/C влево на 3 м”
- результат: план из ~5–10 шагов (`extract common edge -> offset/move -> rebuild -> snap/fix -> validate`).

Оба сценария используют один и тот же API примитивов.

## 12. Нефункциональные требования

- предсказуемость: одинаковый вход -> одинаковый план (при фиксированном профиле);
- наблюдаемость: журнал шагов, метрики, причины ошибок;
- восстановление: rollback + Git snapshot;
- переносимость: сначала локально, затем серверный режим;
- совместимость: Codex/Claude/Antigravity через единый MCP интерфейс.

## 13. План внедрения (этапы)

Этап 0. Hardening форка `qgis_mcp`

- исправить `.sock` -> `.socket`;
- вынести host/port в конфиг;
- добавить базовые тесты и CI;
- добавить лицензию и release-процесс.

Этап 1. Core Engine

- реализовать Plan IR;
- реализовать примитивы и транзакционную обвязку;
- добавить `git_snapshot`.

Этап 2. Validation Engine

- реализовать `cadastre_default` ruleset;
- добавить отчеты по ошибкам валидации.

Этап 3. Planner + Wizard

- `intent_to_plan` с пошаговым уточнением параметров;
- режим preview/apply.

Этап 4. Variants & Comparison

- создание веток вариантов;
- автоматическое сравнение по метрикам.

## 14. Критерии готовности (MVP)

MVP считается готовым, если:

- фразы уровня “раздели на N + дорога” и “сдвинь границу на X м” работают без ручного кода;
- каждая операция проходит через единый валидатор;
- есть rollback и Git snapshot;
- есть preview плана до применения;
- результаты экспортируются в рабочие форматы QGIS.

## 15. Риски и снижения риска

- Риск: нестабильность сокет-соединения.  
  Мера: timeout, heartbeat, retry, message framing.

- Риск: неконсистентные CRS/единицы измерения.  
  Мера: принудительная CRS-проверка до начала плана.

- Риск: тихие геометрические ошибки.  
  Мера: обязательный `fix + topology_validate` до commit.

- Риск: сложные запросы пользователя с недостающими параметрами.  
  Мера: wizard-уточнения и явный preview плана.

## 17. API Contract (Formal)

Нормативный источник контрактов MCP tools:

- `4114/QGIS/MCP-TOOLS-SCHEMA.md`
- `4114/QGIS/schemas/mcp-tools.schema.json`

Правило:

- реализация должна строго соответствовать `api_version=1.0.0`;
- обязательные поля, envelope, error model и совместимость берутся только из контракта.

## 18. Plan IR Spec (Formal)

Нормативный источник Plan IR:

- `4114/QGIS/PLAN-IR-SCHEMA.json`

Правило:

- любой план до исполнения обязан проходить schema validation;
- невалидный план запрещен к исполнению;
- migration-policy для будущих версий обязателен.

## 19. Измеримые критерии MVP Acceptance (SLO/SLA)

Обязательные количественные критерии:

1. SLO preview
- `plan_preview` p95 <= 5 сек для проекта до 10 тыс. объектов.

2. SLO execute
- `plan_execute` p95 <= 120 сек для типового сценария split+road (до 1 тыс. полигонов).

3. SLA reliability
- >= 99% успешных выполнений на валидных входах без ручного вмешательства.

4. Геометрическая точность
- допуск по дистанции <= 0.02 м;
- отклонение площади каждого участка <= 0.1% (если ruleset не задает более строгий допуск).

5. Валидация топологии
- 0 self-intersections, 0 неразрешенных overlaps/gaps на commit.

6. Rollback quality
- >= 99.5% успешных rollback на искусственно инжектированных ошибках.

7. Test coverage gate
- unit coverage >= 80%;
- integration critical paths >= 70%;
- e2e critical scenarios = 100% pass для релиза.
